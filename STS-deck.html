01.1 · 觀察到的現況

既有鐵道應用是「被動式」的
查詢工具，而非旅客的服務代理

現況觀察 · Observations
  • 被動式查詢 — 既有 App (台鐵 e 訂通、台灣高鐵 App) 屬於 Pull-based 工具
  • 整合責任在旅客 — 時刻、月臺、車廂、出口、行事曆、天氣需自行串接
  • 異常成本由旅客承擔 — 誤點、取消、月臺變更皆由旅客自行處置
  • 裝置間斷裂 — 手機 / 手錶 / 看板缺乏狀態同步
問題本質 · Root Cause
「列車為中心」的設計典範，
讓旅客在每一個觸點都必須
執行額外的認知工作。
— Diagnosis · Train-Centric → Passenger-Centric

利害關係人 ① · 採購決策者

對台鐵 / 高鐵管理層 ——
從客訴成本下降切入

痛點 · Pain Points
  • 客訴成本高 — 誤點、找不到車廂、無障礙服務的申訴件數逐年上升
  • 尖離峰運載失衡 — 尖峰滿載、離峰閒置造成資源使用率不佳
  • 數位化資產落差 — 既有票務、調度、影像監控系統各自為政
價值主張 · Value Proposition
  • 客訴下降 — 主動推播 + 即時引導降低 60% 以上「迷路 / 錯過班次」客訴
  • 吞吐提升 — 車廂擁擠度引導，平均化各車廂載客率，提升尖峰月臺淨空率
  • 數據資產化 — 匿名化旅客行為資料 → 動態票價與班次規劃決策依據

利害關係人 ② · 系統操作者

對旅客與乘務員 ——
把認知工作從「人」轉移到「系統」

旅客 · Passenger
  • 不熟悉月臺方向規則、行李動線、車廂編號邏輯
  • 趕車時資訊密度過高，反而增加焦慮與失誤
提供之服務
  • 零認知負荷導航 — 主動推播 + 趕車模式自動切換
  • 動態 QR Code 票券 — 30 秒輪替，離線可用
乘務員 · Crew
  • 無法即時掌握車廂擁擠分布
  • 緊急通報來源座位、特殊需求旅客難以辨識
提供之服務
  • 車廂熱區圖 — 即時擁擠度與上下車人流
  • 精準通報定位 — 緊急來源座位 + 特殊需求標註
將「找路 / 找車廂 / 找出口」的認知工作 轉移至系統後端執行。

利害關係人 ③ · 系統管理者

對營運處與調度中心 ——
讓營運從廣播走向編排

既有痛點
  • 票價結構僵化，無法回應即時供需

  • 動態票價引擎 — 條件式定義離峰折扣 / 尖峰加價 / 團體票
  • 營運儀表板 — 即時車廂熱圖、進站動線、A/B 推播測試
  • 異常事件編排 — 預先定義誤點 / 取消 / 災害自動劇本 (Runbook)

利害關係人 ④ · 安裝維護者

對維修工程師 ——
從「靠經驗」走向「可觀測」

既有痛點
  • 故障診斷靠經驗，缺乏明確錯誤代碼與日誌
  • 設備停機時影響範圍不易評估
  • 維修需中斷整體系統運作
系統提供能力

  • 熱抽換設計 — Beacon、UWB Anchor、月臺顯示器模組化更換
  • 健康檢查 API — 設備自我回報心跳、電量、訊號強度
  • 降級指引 — 故障時系統自動標示影響範圍與替代方案

利害關係人 ⑤ · 關連系統

對外部整合 ——
避免成為新的資料孤島

既有痛點
  • 票務、支付、行事曆、地圖各自獨立，形成資料孤島
  • 缺乏標準化整合介面，新功能上線需逐一客製對接
整合策略

  • 事件匯流排 — 基於 MQTT / Kafka 之發佈訂閱介面
  • 標準化資料模型 — 遵循 GTFS-Realtime、Transmodel
  • 異質運具擴充 — 計程車、共享單車、客運接駁同介面接入

補充 · 預想 Q&A

三題評審委員可能會問的問題

Q1
UWB 全站佈建成本過高，如何說服管理層? A · 採「重點區域優先 + ROI 量化」— 先在客流最大的 5 個車站關鍵月臺佈建，以客訴下降率、運能提升率量化效益，再逐步擴展。
Q2
旅客若不願授權位置，服務是否仍可使用? A · 採「漸進式體驗」— 不授權仍可基本票務查詢;授權位置解鎖主動推播;完整授權 (含行事曆) 啟用智慧出門時間等進階功能。
Q3
5G 通訊或 Beacon 全面失效時，系統如何運作? A · 啟動 L3 最低限度模式 — 票券離線可用、靜態時刻表本地快取、車站既有廣播與看板回退至傳統模式。系統絕不因雲端故障導致全站癱瘓。
